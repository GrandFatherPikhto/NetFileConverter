## 🧩 Как S-выражение превращается в JSON

Рассмотрим пример S-выражения из KiCad:

```lisp
(net 
  (name "GND")
  (node (ref "C1") (pin 1))
  (node (ref "R1") (pin 2))
)
```

В JSON эту же структуру можно представить как:

```json
{
  "type": "net",
  "name": "GND",
  "nodes": [
    { "ref": "C1", "pin": 1 },
    { "ref": "R1", "pin": 2 }
  ]
}
```

Или, если вы хотите полностью сохранить древовидную природу S-выражений, можно использовать более общий подход:

```json
{
  "type": "list",
  "items": [
    { "type": "symbol", "value": "net" },
    { 
      "type": "list",
      "items": [
        { "type": "symbol", "value": "name" },
        { "type": "string", "value": "GND" }
      ]
    },
    {
      "type": "list",
      "items": [
        { "type": "symbol", "value": "node" },
        { "type": "list", "items": [ { "type": "symbol", "value": "ref" }, { "type": "string", "value": "C1" } ] },
        { "type": "list", "items": [ { "type": "symbol", "value": "pin" }, { "type": "number", "value": 1 } ] }
      ]
    }
  ]
}
```

Но, конечно, второй вариант слишком абстрактный и сложный для обработки. **Вы всегда можете спроектировать JSON-схему, которая будет отражать именно те структуры, которые нужны вашей программе.** И это будет гораздо чище и понятнее, чем работать с сырым S-выражением на каждом этапе.

## ✅ Почему JSON лучше для внутреннего представления

- **Чёткая типизация**: JSON различает строки, числа, булевы значения, массивы и объекты. S-выражения всё хранят как списки и атомы, и вам приходится разбирать типы на лету.
- **Удобство десериализации**: В .NET вы можете десериализовать JSON прямо в ваши C#-классы (`NetlistDocument` и т.д.) одной строкой кода. Для S-выражений вам нужно писать кастомный парсер.
- **Человекочитаемость**: JSON гораздо легче читать и править вручную, особенно при отладке.
- **Экосистема**: Огромное количество инструментов для валидации, трансформации и визуализации JSON.

## 🛠️ Как это будет работать в вашем приложении

1. **Парсер KiCad** читает исходный `.net` файл (S-выражения) и преобразует его в **внутреннюю модель `NetlistDocument`** (C#-объекты).
2. **Сериализатор** может сохранить эту модель в JSON-файл (например, `myschema.net.json`) для кеширования или отладки.
3. **Все генераторы** (`_net.txt`, `_bom.txt`, `_dot`, Mermaid) получают на вход уже готовую объектную модель, а не сырые S-выражения.

Таким образом, **S-выражения остаются только на этапе парсинга**, а всё внутреннее взаимодействие строится на JSON-совместимой структуре данных. Это гибко, быстро и легко поддерживать.

Если вы решите добавить поддержку Altium или Eagle в будущем — вам просто нужно будет написать новый парсер, который будет выдавать ту же модель `NetlistDocument`, и всё остальное будет работать без изменений.

**Вывод:** Переход на промежуточный JSON-формат — это идеальное решение, которое не теряет выразительности S-выражений, но выигрывает в простоте и производительности.
