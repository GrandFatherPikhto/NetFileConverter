## Этап 1. Внедрение Dependency Injection и разделение ответственности
**Цель:** сделать класс `Worker` тонким координатором, а всю логику вынести в отдельные сервисы.

### Шаги:
1. **Создать интерфейсы для ключевых компонентов:**
   - `INetlistParser` – парсинг нетлиста в универсальную модель (цепи, компоненты).
   - `IOutputGenerator` – генерация выходных файлов (`_net.txt`, `_bom.txt`, `_dot.txt`, `_orig.txt`).
   - `IFileSystemWatcherFactory` – фабрика для создания `FileSystemWatcher` (чтобы можно было мокать в тестах).
   - `IConfigurationService` – загрузка/сохранение настроек (отделить от UI).
   - `ILogger` – абстракция для логирования.

2. **Реализовать конкретные классы:**
   - `Protel2Parser`, `KiCadParser` – реализуют `INetlistParser`.
   - `NetlistOutputGenerator` – реализует `IOutputGenerator` и использует внутри себя логику генерации каждого файла.
   - `AppConfigurationService` – работает с `appsettings.json` через `Microsoft.Extensions.Configuration` (перестаёт зависеть от WinForms).
   - `FileSystemWatcherWrapper` – обёртка над `FileSystemWatcher` с событиями.

3. **Настроить контейнер DI** (встроенный в .NET Core):
   - В `Program.cs` добавить `Host.CreateDefaultBuilder()` и зарегистрировать все сервисы.
   - `Worker` теперь принимает в конструкторе все зависимости через интерфейсы.

4. **Переписать `Worker`:**
   - Убрать всю логику парсинга и генерации файлов; вызывать сервисы.
   - Обрабатывать события `FileSystemWatcher`, но отправлять имена файлов в очередь (см. Этап 2).

**Результат:** `Worker` стал значительно легче, появилась возможность тестировать каждый сервис изолированно.

## Этап 2. Улучшение надёжности мониторинга файлов
**Цель:** устранить дублирование событий, пропуски и конфликты доступа.

### Шаги:
1. **Внедрить очередь обработки файлов:**
   - Создать `IFileProcessingQueue` (или использовать `Channel<T>` из .NET).
   - В обработчиках событий `FileSystemWatcher` не выполнять сразу обработку, а добавлять путь к файлу в канал с задержкой (например, через `CancellationTokenSource` для сброса таймера при повторных событиях).
   - Отдельный фоновый сервис `FileProcessorService` будет читать из канала и выполнять фактическую обработку с блокировками.

2. **Добавить дедупликацию:**
   - Использовать `ConcurrentDictionary<string, DateTime>` для отслеживания последнего изменения файла.
   - При получении события проверять, прошло ли достаточно времени (например, 500 мс) с последней обработки; если нет – откладывать.

3. **Обработка ошибок доступа:**
   - Ввести механизм повторных попыток (retry) с экспоненциальной задержкой при невозможности открыть файл (потому что он ещё записывается).

**Результат:** приложение перестаёт "дёргаться" на каждое изменение и надёжно обрабатывает даже быстро меняющиеся файлы.

## Этап 3. Разделение на слои и выделение ядра (Core)
**Цель:** отделить бизнес-логику от инфраструктуры и UI.

### Шаги:
1. **Создать сборку `NetFileConverter.Core`:**
   - Перенести туда все интерфейсы, модели данных (цепь, компонент, настройки), абстрактные классы.
   - Никаких ссылок на WinForms, файловую систему (кроме абстракций), конфигурацию.

2. **Создать сборку `NetFileConverter.Infrastructure`:**
   - Реализации парсеров, генераторов, файлового мониторинга, сервиса конфигурации (на основе `Microsoft.Extensions.Configuration`).
   - Зависит от Core.

3. **Сборка `NetFileConverter.App` (или сохранить как `NetFileConverter`):**
   - Содержит `Program.cs`, настройку DI, WinForms-формы и иконку в трее.
   - Зависит от Infrastructure и Core.

4. **Настроить внедрение зависимостей в `Program.cs`:** регистрировать реализации из Infrastructure.

**Результат:** чёткое разделение позволяет легко заменить UI (например, на консоль или WPF) или способ хранения настроек без изменения ядра.

## Этап 4. Улучшение логирования и обработки ошибок
**Цель:** сделать приложение диагностируемым в фоновом режиме.

### Шаги:
1. **Добавить библиотеку Serilog** (или Microsoft.Extensions.Logging) с выводом в файл в `%APPDATA%\NetFileConverter\logs\`.
2. **Внедрить `ILogger<T>` во все сервисы** и добавить информативные логи на ключевые действия (начало/конец обработки, ошибки, предупреждения).
3. **Создать глобальный обработчик исключений** для неперехваченных ошибок (в т.ч. в фоновых потоках) с записью в лог и показом уведомления пользователю (через трей-иконку).

**Результат:** теперь можно легко отследить проблемы по логам, даже если приложение работает незаметно.

**Цель:** отделить форму настроек от бизнес-логики и сделать её более отзывчивой.

### Шаги:
1. **Переписать `SettingsForm`:**
   - Использовать `IConfigurationService` для загрузки/сохранения.
   - Подписаться на изменения конфигурации через `IOptionsMonitor` (или событие), чтобы обновлять отображение при внешнем изменении файла.
   - Убрать прямую работу с JSON.

2. **Добавить валидацию настроек** (например, проверка существования папок) до сохранения.

3. **Вынести логику добавления/удаления папок в отдельный презентер (MVP) или ViewModel (MVVM)** – если планируется переход на более современный UI.

**Результат:** форма становится просто представлением, а бизнес-правила сосредоточены в сервисах.
