## 📊 Форматы для визуализации графов: DOT и альтернативы

**DOT** — действительно **оптимальный и самый практичный выбор** для вашей задачи. Вот почему:

### ✅ Преимущества DOT
- **Стандарт де-факто**: Поддерживается практически всеми инструментами визуализации графов.
- **Богатая экосистема**: Graphviz предоставляет множество алгоритмов компоновки (`dot` — для иерархических графов, `neato` — для симметричных, `fdp`, `circo` и др.).
- **Расширяемость**: Легко добавлять новые атрибуты узлам и рёбрам.
- **Удобочитаемость**: Текстовый формат, понятный человеку.

### 🔄 Альтернативы (и почему они хуже для вашей задачи)

| Формат | Описание | Почему не подходит |
|--------|----------|-------------------|
| **GML** (Graph Modeling Language) | Похож на DOT, но отделяет данные от представления | Менее распространён, меньше инструментов поддержки |
| **GraphML** | XML-формат, богатый и расширяемый | Избыточен для нетлистов, сложнее генерировать и читать |
| **DGML** | XML-формат от Microsoft для ориентированных графов | Узкоспециализирован, привязан к экосистеме Microsoft |
| **GEXF** | Формат для Gephi (анализ сетей) | Перегружен для простой визуализации схем |
| **Mermaid** | Текстовый формат для диаграмм в Markdown | Не поддерживает сложные графы и атрибуты узлов как DOT |
| **JSON** | Универсальный формат данных | Требует отдельного инструмента для рендеринга, нет встроенных алгоритмов компоновки |

### 💡 Вывод
Для вашей задачи **DOT + Graphviz** — **золотой стандарт**. Вы получаете:
- Проверенные алгоритмы компоновки (иерархические схемы отлично смотрятся)
- Возможность экспорта в PNG, SVG, PDF и другие форматы
- Огромное сообщество и документацию

Единственное разумное дополнение — добавить поддержку **Mermaid** как опциональный формат для тех, кто хочет встраивать диаграммы в Markdown-документацию.

## 📝 Обновлённый план рефакторинга (с добавленным Этапом 8)

### Этап 8. Доработка экспорта .dot файлов и визуализации

**Цель:** Сделать генерацию .dot файлов более гибкой, надёжной и удобной для пользователя.

#### Шаги:

1. **Выделить генератор `.dot` в отдельный сервис:**
   - Создать интерфейс `IDotGenerator` с методом `GenerateDot(NetlistModel model, string outputPath)`.
   - Реализовать `DotGenerator`, который принимает модель нетлиста и формирует DOT-файл.
   - Внедрить его через DI в `IOutputGenerator` или вызывать отдельно.

2. **Улучшить качество генерируемых диаграмм:**
   - Добавить атрибуты для улучшения внешнего вида:
     - Форма узлов (`shape=box` для компонентов, `shape=ellipse` для цепей).
     - Цвета для разных типов компонентов (резисторы, конденсаторы, микросхемы).
     - Размер шрифта и отступы для читаемости.
   - Использовать `rankdir=LR` для горизонтальной компоновки (схемы читаются слева направо).
   - Группировать связанные компоненты с помощью `subgraph` (кластеров).

3. **Добавить выбор алгоритма компоновки:**
   - По умолчанию использовать `dot` (иерархический) для направленных графов.
   - Добавить настройку в конфигурации для выбора `neato` (симметричный) или `fdp` (для больших графов).

4. **Сделать генерацию опциональной и устойчивой:**
   - Проверять наличие Graphviz в системе при запуске.
   - Если Graphviz не найден — логировать предупреждение и пропускать генерацию `.dot` (без падения приложения).
   - Добавить настройку в `appsettings.json`: `"GenerateDotFiles": true/false`.

5. **Добавить поддержку Mermaid (опционально):**
   - Создать `IMermaidGenerator` с методом `GenerateMermaid(NetlistModel model)`.
   - Генерировать Mermaid-синтаксис для `flowchart LR` или `graph TD`.
   - Это даст пользователям возможность встраивать диаграммы в GitHub README или документацию.

6. **Написать тесты для генераторов:**
   - Unit-тесты для проверки синтаксиса DOT и Mermaid.
   - Интеграционные тесты с реальными нетлистами.

7. **Обновить документацию:**
   - Описать форматы и способы визуализации в README.
   - Добавить примеры сгенерированных диаграмм.
