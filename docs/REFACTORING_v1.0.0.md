## 🧠 Зачем нужен промежуточный формат?

Текущая архитектура **связывает парсинг и генерацию** в одном конвейере. Если вы захотите:
- добавить новый формат вывода (например, SVG, HTML-отчёт, CSV),
- использовать один и тот же нетлист для нескольких генераций без повторного парсинга,
- вручную править или анализировать данные перед генерацией,
- интегрироваться с другими системами (например, передавать модель в веб-сервис),

то промежуточное представление станет **единым источником истины** и значительно упростит разработку.

## ✅ Выбор формата: JSON vs YAML

| Критерий | JSON | YAML |
|----------|------|------|
| **Читаемость для человека** | Хорошая | Отличная (благодаря отступам) |
| **Компактность** | Компактнее, меньше лишних пробелов | Более раздутый из-за пробелов |
| **Поддержка в .NET** | Встроенная (`System.Text.Json`) | Требуется библиотека (`YamlDotNet`) |
| **Скорость сериализации** | Очень быстрая | Чуть медленнее |
| **Комментарии** | Не поддерживает | Поддерживает (полезно для документирования) |
| **Схемы/валидация** | Можно использовать JSON Schema | Есть аналоги, но менее распространены |

### Мой вердикт:
- **Для внутреннего хранения** (кэш, обмен между модулями) — **JSON**, так как он быстрее, встроен в .NET, и его легко читать машинам.
- **Если вы хотите, чтобы пользователи могли редактировать промежуточные файлы вручную** — тогда **YAML** (благодаря читаемости и комментариям). Но для автоматического использования JSON оптимальнее.

**Рекомендация:** хранить в памяти объектную модель, а при необходимости сохранять её в JSON (например, в папке `out` рядом с другими файлами с расширением `.json`). Это позволит легко загружать данные для повторной генерации без парсинга исходного нетлиста.

## 🧩 Структура модели в памяти

Сейчас, судя по описанию, у вас есть списки цепей и компонентов. Предлагаю унифицировать и расширить модель:

```csharp
public class NetlistDocument
{
    public string SourceFileName { get; set; }       // Исходный .net файл
    public string Format { get; set; }               // "Protel2" или "KiCad"
    public DateTime ParsedAt { get; set; }           // Время парсинга
    public List<Component> Components { get; set; }  // Все компоненты
    public List<Net> Nets { get; set; }              // Все цепи (связи)
    public Dictionary<string, string> Metadata { get; set; } // Доп. информация
}

public class Component
{
    public string Reference { get; set; }      // R1, C2, U3
    public string Value { get; set; }          // 10k, 100nF, ATMega328
    public string Footprint { get; set; }      // 0805, DIP-8
    public string Type { get; set; }           // Резистор, Конденсатор и т.д. (можно вывести из Value)
}

public class Net
{
    public string Name { get; set; }           // GND, VCC, Signal_A
    public List<PinConnection> Pins { get; set; } // Все пины в этой цепи
}

public class PinConnection
{
    public string ComponentRef { get; set; }   // R1
    public string PinNumber { get; set; }      // 1, 2, A, B
}
```

Это позволяет легко генерировать любой выходной формат (BOM, список цепей, граф).

## 🔄 Использование промежуточного формата в конвейере

1. **Парсер** читает исходный нетлист и заполняет `NetlistDocument`.
2. **Сериализатор** (сервис) может сохранить `NetlistDocument` в JSON-файл (опционально, по настройке).
3. **Генераторы** получают `NetlistDocument` и создают `_net.txt`, `_bom.txt`, `_dot` и т.д.
4. Если JSON-файл уже существует и его дата новее исходного нетлиста, можно **пропустить парсинг** и сразу загрузить из JSON (это ускоряет повторную обработку).

## 🛠️ Реализация в рамках рефакторинга

**Добавим новый этап в план:**

### Этап 9. Введение промежуточного представления

**Цель:** Создать универсальную модель данных и сервисы для сериализации/десериализации, отделив парсинг от генерации.

**Шаги:**
1. **Создать модель `NetlistDocument`** и вспомогательные классы в `NetFileConverter.Core`.
2. **Реализовать интерфейс `INetlistSerializer`** с методами `Serialize(NetlistDocument)` и `Deserialize(string)`.
3. **Реализовать `JsonNetlistSerializer`** на основе `System.Text.Json`.
4. **Модифицировать парсеры:** они должны возвращать `NetlistDocument`, а не напрямую генерировать файлы.
5. **Модифицировать `Worker`:** после парсинга опционально сохранять JSON (по настройке), затем передавать `NetlistDocument` в `IOutputGenerator`.
6. **Добавить кэширование:** при повторной обработке того же файла проверять, есть ли свежий JSON, и использовать его вместо повторного парсинга.
7. **Написать тесты** для сериализации/десериализации.

## 💡 Дополнительный совет

Если вы планируете когда-нибудь поддерживать **очень большие нетлисты** (тысячи компонентов), JSON может стать тяжёлым. Тогда можно рассмотреть **MessagePack** или **Protobuf**, но для большинства случаев JSON — идеальный баланс между скоростью, читаемостью и поддержкой.
