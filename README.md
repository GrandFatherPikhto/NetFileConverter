# NetFileConverter — Универсальный конвертер нетлистов (Protel2 / KiCad)

**NetFileConverter** – это легковесное Windows-приложение на базе .NET 8, которое работает в системном трее, отслеживает изменения в выбранных папках и автоматически преобразует файлы нетлистов в упрощённые списки соединений (`_net.txt`), перечни компонентов (`_bom.txt`) и графы для Graphviz (`_net.dot`). Поддерживаются два формата: **Protel2** (Altium) и **KiCad** (s‑expression netlist).

## 🎯 Основное назначение

- **Анализ схем и поиск ошибок** – быстрый просмотр цепей и компонентов без открытия тяжёлого САПР.
- **Общение с нейросетями** – вместо отправки огромных исходников (сотни мегабайт) вы передаёте компактные текстовые файлы `_net.txt` и `_bom.txt`, что ускоряет получение ответа и экономит токены.

---

## 🚀 Функционал

- **Фоновый мониторинг** – `FileSystemWatcher` следит за созданием, изменением и переименованием `.net`-файлов сразу в нескольких папках.
- **Гибкая настройка** – для каждой папки можно указать свой формат (`Protel2` или `KiCad`) прямо в конфигурации или через окно настроек.
- **Динамическая конфигурация** – изменения `appsettings.json` применяются на лету без перезапуска.
- **Уведомления Windows** – всплывающие подсказки информируют об успешной обработке или ошибках.
- **Автоматическое первичное сканирование** – при старте обрабатываются все уже существующие `.net`-файлы.
- **Автоматическая фильтрация мусорных сетей** – при парсинге KiCad сети с именами, содержащими `unconnected` или `no net`, **исключаются** из выходных файлов, чтобы не засорять результат.
- **Интеграция в системный трей** – приложение не занимает место на панели задач, управляется через иконку в трее.
- **Полноценный инсталлятор** – создаётся с помощью Inno Setup; автоматически прописывается в автозапуск, добавляется в `AppPaths` (запуск через `Win+R → NetFileConverter`).

### Что генерируется для каждого `.net`-файла?

Все выходные файлы помещаются в подпапку `out` рядом с исходником:

| Файл | Описание |
|------|----------|
| `*_orig.txt` | Точная копия исходного нетлиста |
| `*_net.txt`  | Список цепей с пинами в формате `Имя_цепи: R1-2, C3-1, ...` (для KiCad сети `unconnected` и `no net` автоматически удалены) |
| `*_bom.txt`  | Перечень компонентов: `Обозначение: Тип (Comment)` |
| `*_net.dot`  | Граф соединений «компонент – цепь» для Graphviz (также без мусорных сетей) |

## 🧱 Архитектура

| Компонент | Назначение |
|-----------|------------|
| `Program.cs` | Точка входа. Настраивает хост (.NET Generic Host), инициализирует иконку в трее, управляет жизненным циклом. |
| `Worker.cs` | Фоновая служба (`BackgroundService`). Реализует `FileSystemWatcher`, парсинг нетлистов (Protel2 / KiCad), генерацию выходных файлов. В KiCad-парсере добавлена фильтрация неиспользуемых сетей. |
| `SettingsForm.cs` | Окно настроек Windows Forms. Позволяет добавлять/удалять папки и выбирать формат для каждой. |
| `appsettings.json` | Конфигурационный файл. Хранит список папок с форматами. Файл находится в `%APPDATA%\NetFileConverter\`, что решает проблемы с правами при установке в `Program Files`. |

### Взаимодействие сервисов (упрощённо)

```
Program.cs → IHost → Worker
Worker → читает appsettings.json → создаёт FileSystemWatcher’ы
FileSystemWatcher → ProcessFile → парсер (Protel2/KiCad) → фильтрация unconnected → запись в out/
SettingsForm → сохраняет настройки → Worker перезагружается через ReloadToken
```

## 💻 Работа с приложением

### Запуск и управление

- После установки приложение автоматически запускается и сворачивается в системный трей.
- **Правая кнопка** по иконке → контекстное меню:
  - *Настройки* – управление отслеживаемыми папками и выбор формата.
  - *Выход* – завершение работы.
- **Двойной клик** по иконке тоже открывает настройки.

### Настройка папок

1. Откройте окно «Настройки».
2. Нажмите **«Добавить папку»** – выберите каталог и **укажите формат** (`Protel2` или `KiCad`).
3. Выберите папку в списке и нажмите **«Удалить»**, чтобы исключить её из мониторинга.
4. Нажмите **«Сохранить»** – настройки запишутся в `%APPDATA%\NetFileConverter\appsettings.json`, и служба автоматически перезапустит наблюдатели.

## 📝 История изменений

### Версия 1.5 — Поддержка Protel Netlist 2.0 и рефакторинг

---

#### `ProtelNetlistParser.cs` — новый файл

Добавлен парсер формата **Protel Netlist 2.0** (Altium Designer).

Формат файла состоит из двух секций:

- **Блоки компонентов** `[...]` — содержат поля `DESIGNATOR`, `FOOTPRINT`,
  `PARTTYPE`, `DESCRIPTION` и др. Строки `*` — разделители пинов, игнорируются.
- **Блоки цепей** `(...)` — первая строка блока — имя цепи, далее узлы вида
  `REF-PIN Value- Passive`.

Узел `B1-104 10CL006YE144C8G- Passive` разбирается по **первому** дефису:
`B1` → designator, `104` → номер пина.

Кодировка определяется автоматически по BOM; если BOM отсутствует —
используется Windows-1252 (стандарт Altium).

---

#### `NetlistProcessor.cs` — переработан

**Добавлено:**

- Метод `DetectFormat(filePath)` — определяет формат по первым 64 байтам файла:
  - `PROTEL NETLIST` → `NetlistFormat.Protel2`
  - `(export` → `NetlistFormat.KiCad`
  - иначе → `NetlistFormat.Unknown` (файл пропускается с записью в лог)
- Перечисление `NetlistFormat { Unknown, KiCad, Protel2 }`
- Диспетчер в `ParseAndSimplify`: вызывает `ParseProtel()` или `ParseKiCad()`
  в зависимости от результата детекции
- Логирование определённого формата: `[Detect] Формат: Protel2`

**Не изменилось:**

- Метод `WriteOutputFiles` — общий для обоих форматов, выходная структура
  файлов одинакова

#### `Worker.cs` — переработан

**Было:** директории захардкожены в коде (`_watchDirectories`), конфиг
игнорировался.

**Стало:** директории читаются из `appsettings.json` через `IConfiguration`,

Структура конфига:

```json
{

    "SourceDirectories": [
      { "Path": "D:\\Projects\\KiCad\\...", "Format": "KiCad"   },
      { "Path": "D:\\Projects\\Altium\\...", "Format": "Protel2" }
    ]
  }
}
```

Поле `Format` в конфиге пока информационное — формат определяется
автоматически по содержимому файла. При добавлении новых форматов
(OrCAD, PADS и др.) поле будет использоваться как подсказка.

#### Формат выходных файлов

При обработке файла `ИМЯ.net` (или `.NET`) создаётся папка `out/` рядом
с исходным файлом. В ней два файла:

---

#### `ИМЯ_net.txt` — упрощённый нетлист

```
=== Упрощённый нетлист для Power_Board.net ===
(Изолированные и unconnected цепи отфильтрованы)

Сеть: +3V3
  └─ C1 (100nF) -> пины: 1, 2
  └─ IC1 (AMS1117-3.3) -> пины: 3
  └─ R5 (10K) [⚠️ ВНИМАНИЕ: НЕТ КОРПУСА] -> пины: 1

Сеть: GND
  └─ C1 (100nF) -> пины: 1
  └─ IC1 (AMS1117-3.3) -> пины: 1, 2
```

**Правила формирования:**

- Цепи отсортированы по имени (алфавитно-лексикографически)
- Цепи с именем, содержащим `unconnected` (в любом регистре), отфильтрованы
- Цепи без узлов (изолированные) не попадают в файл
- Компоненты внутри цепи перечислены в порядке появления в исходном файле
- Пины одного компонента в одной цепи объединены и отсортированы
  численно (9 идёт перед 13)
- При отсутствии футпринта выводится метка `[⚠️ ВНИМАНИЕ: НЕТ КОРПУСА]`

---

#### `ИМЯ_bom.txt` — bill of materials

```
=== BOM (Список компонентов) для Power_Board.net ===

⚠️⚠️⚠️ ВНИМАНИЕ! НАЙДЕНЫ КОМПОНЕНТЫ БЕЗ ПОСАДОЧНЫХ МЕСТ (FOOTPRINT): ⚠️⚠️⚠️
  - R5 (Номинал: 10K)
-------------------------------------------------------------------------

Компонент    | Номинал                    | Корпус (Footprint)
-------------------------------------------------------------------------
C1           | 100nF                      | CAPC1608X90N
C2           | CA45-B-16V-10uF-K         | CAPPM3528X210N
IC1          | AMS1117-3.3               | SOT223
R5           | 10K                        | !!! НЕТ ФУТПРИНТА !!!
```

**Правила формирования:**

- Компоненты отсортированы по designator: сначала по буквенному префиксу
  (C, D, IC, R, ...), затем численно по номеру (R2 перед R10)
- Если хотя бы один компонент без футпринта — в начале файла выводится
  блок предупреждений со списком проблемных компонентов
- Для KiCad: из футпринта вида `Library:FootprintName` берётся только
  `FootprintName` (часть после двоеточия)
- Для Protel: поле `FOOTPRINT` используется как есть

## ⚙️ Примечания для разработчиков

- **Логирование**: при запуске через `dotnet run` или напрямую `.exe` вывод появляется в консоли. В режиме службы (инсталляция) логи можно увидеть через отладчик или перенаправить вывод.
- **Добавление нового формата**: достаточно реализовать метод `ParseXxxNetlist(string[] lines) -> ParseResult` и добавить условие в `ProcessFile`.
- **Фильтрация сетей**: если вы хотите добавить другие шаблоны для исключения (например, `"NC"`), отредактируйте условие в методе `ParseKiCadNetlist` в `Worker.cs`.
- **Путь к конфигурации**: `Environment.GetFolderPath(Environment.SpecialFolder.ApplicationData) + "\\NetFileConverter\\appsettings.json"`.
- **Автозапуск**: обеспечивается ярлыком в `C:\ProgramData\Microsoft\Windows\Start Menu\Programs\StartUp`. Удаляется автоматически при деинсталляции.

## ❓ Часто задаваемые вопросы

**Q: Где лежат настройки, если программа установлена в Program Files?**  
A: В `%APPDATA%\NetFileConverter\appsettings.json`. Вы можете открыть его в любом текстовом редакторе.

**Q: Какой формат KiCad поддерживается?**  
A: Экспорт схемы в формате `(kicad_netlist ...)` – любой версии от 6 до 10. Используйте `File → Export → Netlist` в KiCad и выберите формат «KiCad».

**Q: Можно ли добавить папку уже после того, как программа запущена?**  
A: Да, через окно настроек. Настройки применяются без перезапуска.

**Q: Почему не создаётся папка `out`?**  
A: Проверьте, что файл имеет расширение `.net` (регистр не важен), и что формат указан правильно (Protel2 / KiCad). Посмотрите логи в консоли, если запускаете вручную.

**Q: В выходных файлах всё ещё видны сети `unconnected`. Что делать?**  
A: Убедитесь, что вы используете версию **1.3** или новее. Если вы собрали приложение из исходников – проверьте, что в `Worker.cs` в методе `ParseKiCadNetlist` присутствует фильтрация по `Contains("unconnected")`. Если нет – обновите файл из репозитория.
